# Demo 102 v2: Real-World Progressive Disclosure — Design

**Date:** 2026-06-19
**Demo dir:** `102-ent-tokenomics/` (extends v1)
**Status:** Design for review
**Builds on:** `2026-06-19-progressive-disclosure-search-mode-design.md` (v1, shipped & deployed)

## Goal

Make the search-mode savings demo credible to a skeptical enterprise buyer by
moving past the synthetic single-tool proof. Five additions, all locked with the
user:

1. **Honest tradeoff metrics** — search/code modes aren't free; measure the full
   picture: LLM round-trips, wall-clock latency (p50/p95), net total tokens, and
   task-success rate per mode.
2. **All four tool modes** — Standard / Search / Code / CodeSearch (the full CRD
   enum), not just on/off.
3. **Real cache-aware costing** — capture actual cache token usage from both
   OpenAI (automatic, `cached_tokens`, 50% off, ≥1024-tok floor) and Anthropic
   (explicit `cache_control`, `cache_creation_input_tokens` @1.25×,
   `cache_read_input_tokens` @0.1×). Prove search mode still wins *with caching on*.
4. **Business cost projection** — translate per-task numbers into $/month at
   realistic volumes (10k / 50k / 200k agent calls/day) across both models.
5. **Showcase Solo UI tracing** — surface the `get_tool` / `invoke_tool` /
   `run_code` spans (tracing was fixed in v1.x; this makes it a visible feature).

## Spike findings (verified live, 2026-06-19)

All four modes function against the synthetic SSE server through the gateway:

| Mode | Advertised tools | Model interaction |
|------|------------------|-------------------|
| Standard | N (all) | Full schema of every tool |
| Search | 2 | `get_tool` (look up) + `invoke_tool` (call by name) |
| Code | 1 | `run_code` — model writes JavaScript that orchestrates tools |
| CodeSearch | 2 | `get_tool` + `run_code` (discover, then code-execute) |

Implication: **Code/CodeSearch can collapse multi-tool tasks into one round-trip**
(a single code block calls many tools), which is exactly the tradeoff dimension #1
must measure — fewer advertised tools *and* potentially fewer round-trips, at the
cost of code-gen completion tokens and execution risk.

Tracing fix already shipped: an `EnterpriseAgentgatewayPolicy` (`frontend.tracing`)
points the proxy OTLP exporter at `solo-enterprise-telemetry-collector:4317`;
verified spans land in `platformdb.otel_traces_json`. Now in `deploy.sh` Step 5b.

## Decisions (locked)

| Decision | Choice |
|----------|--------|
| Models | OpenAI `gpt-4o-mini` (existing) **+** Anthropic `claude-sonnet-4-6` (new backend) |
| Caching data | **Real** on both: OpenAI `cached_tokens`; Anthropic `cache_creation`/`cache_read` |
| Modes | All 4: Standard, Search, Code, CodeSearch |
| Cache measurement | Each task run **cold then warm** (back-to-back) to capture write vs read |
| Tradeoff metrics | round-trips (LLM calls), latency p50/p95, net total tokens, task_ok rate |
| Dashboards | Keep the v1 headline dashboard **simple**; add a separate **Deep-Dive** dashboard for tradeoff/cache/modes/projection |
| Secret handling | `ANTHROPIC_API_KEY` via gitignored `.env` → Secret (sed), never committed |

## Architecture changes

```
 agentgateway-proxy
  ├─ /mcp/<mode>-<count>   mode ∈ {standard, search, code, codesearch}, count ∈ {10,50,100}
  │                         → EnterpriseAgentgatewayBackend (12 backends, 12 routes)
  ├─ /openai               → AgentgatewayBackend (openai gpt-4o-mini)        [v1]
  └─ /anthropic            → AgentgatewayBackend (anthropic claude-sonnet-4-6) [NEW]

 Harness (provider-agnostic):
   for provider in {openai, anthropic}:
     for mode in {standard, search, code, codesearch}:
       for count in {10,50,100}:
         run task COLD (cache miss) then WARM (cache hit), R times
         → capture usage incl. cache fields, llm_calls, latency
   → results.csv/json  → Pushgateway  → Prometheus  → Grafana (2 dashboards)
   → projection.py: $/month at volume × model × mode × cache-state
```

Note: 12 MCP backends (4 modes × 3 counts) generated by the existing `deploy.sh`
loop (extend the mode list from `default search` to `standard search code codesearch`,
mapping names → toolMode enum `Standard/Search/Code/CodeSearch`).

## Components

### 1. Anthropic LLM backend (`k8s/anthropic.yaml` + deploy.sh)
`AgentgatewayBackend` with `provider.anthropic`, model `claude-sonnet-4-6`,
auth from `anthropic-secret` (sed-substituted `ANTHROPIC_API_KEY`), route `/anthropic`.
deploy.sh preflight gains an `ANTHROPIC_API_KEY` check.

**RISK / first implementation task:** confirm how the AGW anthropic backend
presents its API (OpenAI-compatible `/chat/completions` vs Anthropic-native
`/v1/messages`) and how `tools` + `cache_control` + `usage` round-trip. This is
unverified and must be spiked live before building the adapter. The plan's first
Anthropic task is a probe, not an assumption.

### 2. Harness v2 (`harness/run_ab.py` refactor + `harness/providers.py`)
- **Provider adapter** (`providers.py`): per provider, build the request (model,
  messages, tools, cache breakpoints), parse `tool_calls`, and extract a normalized
  `usage` dict: `prompt_tokens, completion_tokens, cached_tokens,
  cache_write_tokens, cache_read_tokens`.
- **Cold/warm runs**: execute each task twice back-to-back; tag rows `cache_state ∈ {cold, warm}`.
- **Metrics captured per row**: `provider, model, mode, tool_count, run, cache_state,
  advertised_tools, first_call_prompt_tokens, total_prompt_tokens, completion_tokens,
  cached_tokens, cache_write_tokens, cache_read_tokens, total_tokens, llm_calls,
  latency_ms, usd_cost_uncached, usd_cost_cached, task_ok`.
- **Realistic task**: upgrade from "call tool_007" to a small multi-step task that
  requires using 2–3 tools, so Code mode's round-trip collapse is observable. Tools
  stay deterministic echoes; the task references specific tool names + args so
  success is checkable across all 4 modes.

### 3. Pricing model (`harness/pricing.json`)
Per provider/model: `input_per_1k, cached_input_per_1k` (OpenAI),
`cache_write_per_1k, cache_read_per_1k` (Anthropic), `output_per_1k`. Cost fns
compute both uncached and cache-aware cost from the real token breakdown.

### 4. Cost projection (`harness/projection.py`)
Reads `results.csv`, emits `projection.csv` + console table: for each
(provider, mode, cache_state), $/day and $/month at 10k/50k/200k calls/day, plus
$ saved vs Standard. Pushes projection gauges for Grafana.

### 5. Dashboards (`observability/`)
- **`dashboard.json`** (v1, keep simple): the headline savings curve — unchanged
  except it now filters to one provider/mode pair so it stays uncluttered.
- **`dashboard-deepdive.json`** (NEW): sections — (a) 4-mode token comparison,
  (b) round-trips & latency by mode, (c) cache impact (cost cold vs warm, cached vs
  uncached), (d) $/month projection by volume. Each panel keeps a plain-language
  description. Provisioned via a second ConfigMap labeled `grafana_dashboard=1`.

## Testing strategy
- Determinism where possible: `temperature=0`, seeds; echo tools; R≥5.
- Cold/warm correctness: assert warm runs show `cached_tokens>0` (OpenAI, ≥1024-tok
  prompts) / `cache_read_tokens>0` (Anthropic); cold runs show cache writes/none.
- Cross-mode sanity: advertised tools == {N,2,1,2} for {Standard,Search,Code,CodeSearch}.
- Per-mode `task_ok` recorded (Code mode completion is data, not a gate).
- Offline checks as in v1 (py_compile, json/yaml, bash -n); live sweep produces proof.

## Out of scope (YAGNI)
- Real third-party MCP servers — synthetic knob still gives the cleanest curve;
  revisit only if the token magnitudes are questioned.
- Authorization-based tool filtering — separate concern; not measured here.
- Metrics→ClickHouse pipeline for the Solo UI — tracing is enabled; UI metrics is a
  separate toggle and not required for the proof.
- More than two models — two frontier vendors is enough to prove generality.

## Open risks (addressed in the plan, not assumed away)
1. **Anthropic API shape through AGW** — spike first (Component 1).
2. **Code/CodeSearch task completion** — the model must write working JS that calls
   echo tools; if `task_ok` is low, we report it honestly (it's a real finding about
   code-mode maturity), and the token/latency comparison still stands.
3. **Warm-cache timing** — cache TTL/threshold; cold/warm back-to-back mitigates TTL,
   and OpenAI's 1024-tok floor means small search/code prompts may never cache (which
   is itself the point — we report it).
